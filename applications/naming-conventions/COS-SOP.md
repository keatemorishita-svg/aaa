# COS-SOP

## How this was made

1. **需求识别**: 大量论文文件需要按"代码可用性"和"已读/未读"状态快速过滤
2. **约束发现**: Obsidian 的文件列表按字符串排序——这从约束变成了特性
3. **信息编码**: 将三维元数据压缩到文件名前缀，利用排序实现优先级排列
4. **渐进自动化**: 先自动加 G（有 GitHub 仓库），用户手动加 A（已解构），不增加初始标注成本

## Key Decisions

1. **倒序分而非正序分** — 字符串升序排列下，高分需要小的数值。这是从文件管理器的机械行为反向推导的设计
2. **G/A 而非完整单词** — 单字符标记使文件名保持可读，同时携带最大信息密度
3. **v3 而非首次设计** — 前两个版本被放弃，因为标记位过多（v1）或排序逻辑不对（v2）

## What you tried and abandoned

- v1: 全标记体系（G/A/R/T/S 五个标记位）——过于复杂，日常使用中 80% 的标记位从未被使用
- v2: 正序分（分数本身放文件名）——排序后发现低分文件排前面，违背直觉
- 结论：一个好的协议是通过删除而不是添加来达到最终形式的
