# Tool: Three-Primitive System Diagnostic

> Standalone diagnostic. Copy, use, share. No AAA knowledge required.

---

## What this is

A 5-minute diagnostic to identify why any system (organization, project, community) is failing to evolve.

---

## The Diagnostic

For each primitive, score 1-5 (1 = broken, 5 = healthy).

### ACCESS — Can new people and ideas enter?

| Score | Description |
|---|---|
| 1 | Nobody new can enter. System is sealed. |
| 2 | Entry is possible but extremely difficult (months of onboarding, gatekeepers). |
| 3 | Entry is moderately difficult (some friction, documentation exists but incomplete). |
| 4 | Entry is easy for most (clear paths, documentation, welcoming culture). |
| 5 | Entry is frictionless. Anyone can participate immediately. |

### MUTATION — Can the system be changed by its participants?

| Score | Description |
|---|---|
| 1 | Nothing can be changed. System is frozen. |
| 2 | Change is technically possible but practically impossible (requires top-level approval, 6-month process). |
| 3 | Change is possible with effort (defined process, takes weeks, needs multiple approvals). |
| 4 | Change is straightforward (clear process, takes days, one approval). |
| 5 | Anyone can change anything at any time. Zero friction. |

**Your score**: ___  
**Evidence** (be specific): _______________________________

---

### RECURSION — Can the rules for changing the system be changed?

| Score | Description |
|---|---|
| 1 | Rules are immutable. The system cannot examine or modify its own processes. |
| 2 | Rules can theoretically change but require the same authority that created them. |
| 3 | Rules can change through a defined process (annual review, committee vote). |
| 4 | Rules can change easily (any participant can propose; lightweight approval). |
| 5 | The system continuously adapts its own rules. Meta-evolution is automatic. |

---

## Interpretation

| Score Pattern | Diagnosis | Prescription |
|---|---|---|
| All ≥ 4 | System is healthy. Evolution is possible. | Maintain. Watch for creeping centralization. |
| One primitive < 3 | System has a specific blockage. | Fix that primitive. Don't reorganize everything. |
| Access < 3 | System is a black box or walled garden. | Publish documentation. Create onboarding. Open the doors. |
| Mutation < 3 | System is bureaucratic. Change is too hard. | Simplify change process. Reduce approvals. Empower edges. |
| Recursion < 3 | System is a dictatorship or frozen bureaucracy. | Create a process for changing processes. Make governance transparent. |
| All < 3 | System is closed, frozen, and authoritarian. | The system is not evolving. It's dying. Radical intervention or exit. |
| Access 5 + Mutation 5 + Recursion 1 | **Danger**: Open and changeable but no integration. Fragments into chaos. | Build merge mechanisms. Define selection criteria. |

---

## Example

**System diagnosed**: A 200-person engineering organization.

- **Access 2/5**: New hires take 3 months to get commit access. External contributors have no path in.
- **Mutation 2/5**: Any architecture change requires VP approval. Average change takes 6 weeks.
- **Recursion 1/5**: The VP who designed the approval process is the only one who can change it.

**Diagnosis**: All three primitives are broken. The system is a frozen bureaucracy. The prescription is not "better tools" or "more Agile coaching" — it's Access first (open the doors), then Mutation (reduce approval chain), then Recursion (let the team change its own process).
